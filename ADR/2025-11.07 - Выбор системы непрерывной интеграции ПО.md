**Дата:** 7 ноября 2025 г.

**Статус:** Принято

### **Контекст**

Для автоматизации процессов сборки, тестирования и развертывания приложения на платформе для поиска и резервирования билетов на концерты необходимо выбрать систему CI/CD. Этот выбор будет влиять на скорость разработки, качество кода и общую эффективность команды.

### **Рассмотренные варианты**

1. **GitLab CI/CD:** Интегрированная система CI/CD, которая работает непосредственно с репозиториями GitLab и предлагает мощные инструменты для автоматизации.
2. **Jenkins:** Популярный инструмент с открытым исходным кодом, который предоставляет гибкость и множество плагинов для настройки CI/CD процессов.
3. **GitHub Actions:** Инструмент для автоматизации рабочих процессов, интегрированный с GitHub, позволяющий легко настраивать CI/CD для проектов, размещенных на этой платформе.
4. **CircleCI:** Облачная платформа CI/CD, которая предлагает простоту настройки и высокую скорость выполнения задач.

1. **GitLab CI/CD**
   **Плюсы:**

   - Полная интеграция с репозиториями GitLab.
   - Простая конфигурация через .gitlab-ci.yml.
   - Поддержка параллельных и распределенных пайплайнов.
   - Встроенные средства мониторинга и уведомлений.
   - Возможность использования собственной инфраструктуры или облака.
   

   **Минусы:**

   - Требуется развертывание и администрирование собственной среды (если не используют облачный сервис).
   - Меньше плагинов по сравнению с Jenkins.
   - Зависимость от экосистемы GitLab.

2. **Jenkins**
   **Плюсы:**

   - Высокая гибкость благодаря тысячам доступных плагинов.
   - Поддержка множества языков и платформ.
   - Возможность развертывания на собственной инфраструктуре.
   - Хорошая поддержка скриптов и кастомизации.

   **Минусы:**

   - Требует настроек и поддержки собственной инфраструктуры.
   - Могут возникать сложности с управлением большими пайплайнами.
   - Настройка и обновление могут быть сложными для новичков.


3. **GitHub Actions**
   **Плюсы:**

   - Легкая интеграция с репозиториями на GitHub.
   - Простое создание рабочих процессов через YAML.
   - Мгновенно запускается и легко настраивается.


   **Минусы:**

   - Можно использовать преимущественно только внутри экосистемы GitHub.
   - Меньше расширенных функций по сравнению с Jenkins или GitLab.
   - Ограничения по времени выполнения для бесплатных аккаунтов.
   

4. **CircleCI**
   **Плюсы:**

   - Простая настройка и быстрый запуск.
   - Хорошая масштабируемость и облачная инфраструктура.
   - Поддержка Docker и Kubernetes.
   - Гибкий тарифный план, включая бесплатный.

   **Минусы:**

   - Меньше возможностей для кастомизации по сравнению с Jenkins.
   - Требуется подключение сторонних сервисов для расширенной логики.
   - В случае необходимости — дополнительная сложность в настройке корпоративной инфраструктуры.

### **Решение**

Выбран **GitLab CI/CD**

### **Обоснование**

- Интеграция с существующим репозиторием и рабочими процессами GitLab позволяет минимизировать время на настройку и повысить эффективность.
- Наличие мощных инструментов автоматизации и мониторинга обеспечивает быстрый цикл внедрения и высокое качество релизов.
- Возможность использования собственной инфраструктуры или облака дает гибкость в масштабировании и обеспечении безопасности.
- Простота конфигурации через файл .gitlab-ci.yml способствует быстрой реализации процессов автоматической сборки и тестирования.
- В случае дальнейшего расширения проекта и необходимости интеграции с DevOps-инструментами GitLab является оптимальным выбором.


---

### **Последствия**

- Требуется обучение команды работе с GitLab Runner и созданию эффективных пайплайнов.
- Потребуется интеграция с системами мониторинга для отслеживания состояния пайплайнов и быстрого реагирования на сбои.
- При необходимости развертывания собственной инфраструктуры потребуется настройка серверов и конфигурация Runner-ов.
- Внедрение позволит сократить цикл релиза, повысить стабильность выпускаемого программного обеспечения и снизить ручной труд при деплое.
